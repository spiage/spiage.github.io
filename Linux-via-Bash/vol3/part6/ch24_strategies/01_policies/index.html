    Политики хранения: Full, Incremental, Differential, Snapshot¶

    Тип бэкапа -- это фундаментальное решение, которое влияет на три ключевых параметра:

    1. Объём хранилища -- сколько места занимают резервные копии
    2. Скорость создания -- как долго длится процесс бэкапа
    3. Скорость восстановления -- сколько времени нужно, чтобы вернуть данные

    Не существует «лучшего» типа -- есть оптимальный для конкретной ситуации. Разберём каждый.

    Full backup (полный бэкап)¶

    Full backup -- это полная копия всех выбранных данных. Каждый раз, когда запускается полный бэкап, копируется всё целиком, независимо от того, менялись файлы или нет.

    # Пример полного бэкапа через tar
$ sudo tar -czpf /backup/full-$(date +%F).tar.gz \
    --preserve-permissions \
    --same-owner \
    /home /etc /var/www

    Характеристики¶

    Параметр Значение
    Объём Максимальный -- полная копия всех данных
    Скорость создания Самая медленная
    Скорость восстановления Самая быстрая -- нужен только один файл
    Зависимости Нет -- автономный архив

    Когда использовать¶

    • Первый бэкап в цепочке (база для последующих инкрементальных)
    • Критичные данные небольшого объёма
    • Периодические «якорные» копии (раз в неделю/месяц)
    • Когда простота восстановления важнее экономии места

    Важно: Full backup занимает больше всего места среди всех типов. Если у вас 500 ГБ данных и вы делаете полный бэкап каждый день, за неделю потребуется 3,5 ТБ -- без учёта сжатия.

    Incremental backup (инкрементальный)¶

    Incremental backup копирует только те файлы, которые изменились с последнего бэкапа любого типа. Первый инкрементальный бэкап всегда идёт после полного.

    # Пример инкрементального бэкапа через tar
# Сначала -- полный бэкап (воскресенье)
$ sudo tar -czpf /backup/full-$(date +%F).tar.gz \
    --listed-incremental=/backup/snapshot.snar \
    /home

# Далее -- инкрементальные (понедельник-суббота)
$ sudo tar -czpf /backup/incr-$(date +%F).tar.gz \
    --listed-incremental=/backup/snapshot.snar \
    /home

    Флаг --listed-incremental заставляет tar вести файл-снимок (snapshot.snar), в котором хранятся метаданные о состоянии файлов на момент последнего бэкапа.

    Характеристики¶

    Параметр Значение
    Объём Минимальный -- только изменения
    Скорость создания Самая быстрая
    Скорость восстановления Самая медленная -- нужна цепочка
    Зависимости Требует Full + все инкременты

    Цепочка восстановления¶

    Для восстановления из инкрементальных бэкапов нужна вся цепочка:

    Full (Вс) → Incr (Пн) → Incr (Вт) → Incr (Ср)

    Чтобы восстановить данные на среду, нужно последовательно применить: 1. Full backup (Вс) 2. Incremental (Пн) 3. Incremental (Вт) 4. Incremental (Ср)

    Если хотя бы один инкремент в цепочке повреждён -- восстановление после него невозможно.

    Когда использовать¶

    • Ограниченное хранилище
    • Частые бэкапы (каждый час/день)
    • Данные меняются незначительно между бэкапами
    • Есть надёжная система хранения цепочки

    Differential backup (дифференциальный)¶

    Differential backup копирует изменения с последнего полного (Full) бэкапа. В отличие от инкрементального, каждый дифференциальный бэкап растёт в размере, но не зависит от предыдущих дифференциальных.

    # Пример дифференциального бэкапа
# Полный бэкап (воскресенье)
$ sudo tar -czpf /backup/full-$(date +%F).tar.gz /home

# Дифференциальный (понедельник) -- изменения с воскресенья
$ sudo tar -czpf /backup/diff-$(date +%F).tar.gz \
    --newer=/backup/full-2026-04-01.tar.gz \
    /home

# Дифференциальный (вторник) -- тоже изменения с воскресенья
$ sudo tar -czpf /backup/diff-$(date +%F).tar.gz \
    --newer=/backup/full-2026-04-01.tar.gz \
    /home

    Характеристики¶

    Параметр Значение
    Объём Средний -- растёт с каждым бэкапом
    Скорость создания Средняя
    Скорость восстановления Быстрее инкрементального -- нужны только Full + последний Differential
    Зависимости Требует Full + последний Differential

    Цепочка восстановления¶

    Full (Вс) → Diff (Ср)  -- достаточно двух файлов

    Для восстановления на любую дату нужны только два файла: полный бэкап и последний дифференциальный. Это компромисс между Full и Incremental.

    Snapshot (снапшот)¶

    Snapshot -- это мгновенный «снимок» файловой системы на определённый момент времени. В отличие от предыдущих типов, снапшот не копирует данные -- он фиксирует состояние метаданных.

    LVM снапшоты¶

    # Создание LVM снапшота
$ sudo lvcreate --size 10G --snapshot --name home_snap /dev/vg0/home

# Монтирование снапшота для бэкапа
$ sudo mount /dev/vg0/home_snap /mnt/snapshot

# Бэкап со снапшота (система продолжает работать)
$ sudo tar -czpf /backup/from-snapshot-$(date +%F).tar.gz /mnt/snapshot/

# Удаление снапшота
$ sudo lvremove /dev/vg0/home_snap

    Btrfs снапшоты¶

    # Создание снапшота btrfs
$ sudo btrfs subvolume snapshot /home /home/.snapshots/$(date +%F)

# Снапшот занимает почти 0 места (copy-on-write)
$ sudo btrfs subvolume list /home

    ZFS снапшоты¶

    # Создание снапшота ZFS
$ sudo zfs snapshot tank/home@$(date +%F)

# Просмотр снапшотов
$ sudo zfs list -t snapshot

# Восстановление отдельного файла из снапшота
$ cp /tank/home/.zfs/snapshot/2026-04-01/important.txt /home/user/

    Характеристики снапшотов¶

    Параметр Значение
    Объём Минимальный на создание (только метаданные)
    Скорость создания Мгновенная
    Скорость восстановления Мгновенная (rollback) или копирование файлов
    Зависимости Зависит от поддержки ФС (btrfs, ZFS, LVM)

    Важно: Снапшот -- это не бэкап в классическом смысле. Он хранится на том же пуле дисков. Если диск выйдет из строя, снапшот тоже будет потерян. Снапшоты нужно копировать на отдельный носитель для полноценной защиты.

    Сравнительная таблица¶

    Критерий Full Incremental Differential Snapshot
    Объём Максимальный Минимальный Средний (растёт) Минимальный
    Скорость бэкапа Медленная Быстрая Средняя Мгновенная
    Скорость восстановления Быстрая Медленная Средняя Быстрая
    Файлов для восстановления 1 Full + все инкременты Full + последний diff 1 (rollback)
    Риск потери цепочки Нет Высокий Средний Низкий
    Требуемая ФС Любая Любая Любая LVM/btrfs/ZFS

    Как выбрать тип бэкапа¶

    Сценарий 1: Домашний ноутбук (100-500 ГБ)¶

    Full: раз в неделю
Incremental: ежедневно

    Мало данных, частые изменения, ограниченное хранилище.

    Сценарий 2: Веб-сервер (1-10 ТБ)¶

    Full: раз в месяц
Differential: ежедневно
Snapshot: перед каждым деплоем

    Большие данные, важна скорость восстановления, снапшоты для отката деплоя.

    Сценарий 3: База данных (10+ ТБ)¶

    Full: раз в неделю
Incremental: каждые 15 минут (WAL-логи)
Snapshot: перед обслуживанием

    Критичные данные, минимальная потеря данных (RPO), снапшоты для безопасных операций.

    Практический совет: измерение изменений¶

    Прежде чем выбрать тип бэкапа, измерьте, сколько данных реально меняется:

    # Найти файлы, изменённые за последние 24 часа
$ find /home -type f -mtime -1 -print0 | du -ch --files0-from=- | tail -1

# Сравнить с общим объёмом
$ du -sh /home

    Если за день меняется менее 5-10% данных -- инкрементальный бэкап будет эффективен. Если 30-50% -- разница между типами менее заметна.
