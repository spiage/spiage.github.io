    ACL и расширенные атрибуты (xattr)¶

    Стандартных прав rwx (владелец/группа/остальные) часто недостаточно. Что если нужно дать доступ к файлу конкретному пользователю, не входящему в группу? Или прикрепить к файлу произвольные метаданные? Для этого существуют ACL и расширенные атрибуты. При бэкапе их легко потерять -- и это одна из самых коварных проблем восстановления.

    Что такое ACL¶

    ACL (Access Control List) -- расширенный список контроля доступа, позволяющий задавать права для конкретных пользователей и групп, помимо стандартных rwx.

    # Стандартные права: только владелец, группа, остальные
$ ls -l /srv/project/data.txt
-rw-r----- 1 admin developers 4096 мар 15 10:00 /srv/project/data.txt

# Нужно дать доступ пользователю user1 (не в группе developers)
# Без ACL пришлось бы добавить user1 в группу -- но это лишние права

    Просмотр ACL¶

    # getfacl -- показывает ACL файла
$ getfacl /srv/project/data.txt
# file: srv/project/data.txt
# owner: admin
# group: developers
user::rw-
user:user1:r--          # <-- конкретный пользователь
group::r--
group:testers:r-x       # <-- конкретная группа
mask::r-x
other::---

    getfacl показывает полный ACL. Команда ls -l показывает только стандартные права и наличие ACL обозначает символом +:

    $ ls -l /srv/project/data.txt
-rw-r-x---+ 1 admin developers 4096 мар 15 10:00 /srv/project/data.txt
#          ^
#          '+' означает наличие ACL

    Управление ACL¶

    # setfacl -- установить ACL
$ sudo setfacl -m u:user1:r-- /srv/project/data.txt

# Дать группе права на чтение и выполнение
$ sudo setfacl -m g:testers:r-x /srv/project/data.txt

# Удалить запись ACL
$ sudo setfacl -x u:user1 /srv/project/data.txt

# Рекурсивно установить ACL на каталог
$ sudo setfacl -R -m u:user1:r-x /srv/project/

# Default ACL: новые файлы в каталоге наследуют права
$ sudo setfacl -d -m u:user1:r-x /srv/project/

    Расширенные атрибуты (xattr)¶

    xattr (extended attributes) -- механизм для хранения произвольных метаданных, привязанных к файлу. Это пары «ключ-значение», которые не входят в стандартные метаданные inode.

    # Посмотреть расширенные атрибуты
$ getfattr -d /path/to/file

# Примеры использования xattr:
# - Метки безопасности SELinux/AppArmor
# - Атрибуты капabilities
# - Пользовательские метаданные

    Практические примеры xattr¶

    # SELinux контекст (хранится как xattr)
$ getfattr -n security.selinux /etc/passwd
# file: etc/passwd
security.selinux="system_u:object_r:etc_t:s0"

# Пользовательские атрибуты
$ setfattr -n user.comment -v "important config" /etc/myapp.conf
$ getfattr -n user.comment /etc/myapp.conf
# file: etc/myapp.conf
user.comment="important config"
    Префикс xattr Назначение
    user.* Пользовательские атрибуты
    system.* Системные атрибуты (ACL, capabilities)
    security.* Метки безопасности (SELinux, AppArmor)
    trusted.* Только для root

    Риски потери ACL и xattr при бэкапе¶

    cp по умолчанию НЕ сохраняет ACL¶

    # cp без флагов: ACL теряется!
$ cp /srv/project/data.txt /backup/data.txt
$ getfacl /backup/data.txt
# file: backup/data.txt
# owner: user
# group: user
user::rw-
group::r--
other::---
# ACL записей для user1 и testers НЕТ!

    # cp с флагом --preserve=acl или -a: ACL сохраняется
$ cp --preserve=acl /srv/project/data.txt /backup/data.txt
# или
$ cp -a /srv/project/data.txt /backup/data.txt

    rsync: флаги для ACL и xattr¶

    # Без -AX: ACL и xattr потеряются
$ rsync -a /srv/project/ /backup/project/
# ACL и xattr НЕ сохранятся

# С -AX: полное сохранение
$ sudo rsync -aAX /srv/project/ /backup/project/
#         ^^
#         -A (--acls):  сохранить ACL
#         -X (--xattrs): сохранить xattr
    Флаг Длинная форма Что делает
    -A --acls Сохраняет Access Control Lists
    -X --xattrs Сохраняет extended attributes

    Без флагов -A и -X файлы восстановятся, но ACL и расширенные атрибуты будут потеряны. Это может привести к тому, что пользователи потеряют доступ к файлам, а системы безопасности (SELinux) перестанут работать корректно.

    Полный бэкап с сохранением всех атрибутов¶

    # tar: сохранение ACL и xattr
$ sudo tar -czpf backup.tar.gz \
    --acls \
    --xattrs \
    --same-owner \
    -p \
    /srv/project/

# rsync: полный бэкап
$ sudo rsync -aAXv \
    /source/ \
    /backup/

    Проверка после восстановления¶

    # Сравнить ACL до и после
$ getfacl /srv/project/data.txt > /tmp/acl-before.txt
# ... восстановление ...
$ getfacl /backup/data.txt > /tmp/acl-after.txt
$ diff /tmp/acl-before.txt /tmp/acl-after.txt

# Сравнить xattr
$ getfattr -d -R /srv/project/ > /tmp/xattr-before.txt
# ... восстановление ...
$ getfattr -d -R /backup/ > /tmp/xattr-after.txt
$ diff /tmp/xattr-before.txt /tmp/xattr-after.txt

    Практический сценарий: бэкап веб-сервера¶

    #!/bin/bash
# backup-webapp.sh -- бэкап с сохранением ACL и xattr

SOURCE="/var/www/myapp"
BACKUP="/backup/webapp-$(date +%F).tar.gz"

sudo tar -czpf "$BACKUP" \
    --acls \
    --xattrs \
    --preserve-permissions \
    --same-owner \
    --xattrs-include='*' \
    -C / \
    var/www/myapp

echo "Бэкап создан: $BACKUP"

# Проверка
$ tar -tzf "$BACKUP" | head -5
$ tar --list --acls -f "$BACKUP" | head -20

    Резюме¶

    • ACL (Access Control List) -- расширенный список доступа для конкретных пользователей/групп
    • xattr (extended attributes) -- произвольные метаданные (SELinux, capabilities, пользовательские атрибуты)
    • cp по умолчанию НЕ сохраняет ACL -- нужен --preserve=acl или -a
    • getfacl -- показывает ACL файла
    • rsync -A (--acls) и -X (--xattrs) -- сохраняют ACL и xattr
    • Без -AX файлы восстановятся, но ACL/xattr будут потеряны
