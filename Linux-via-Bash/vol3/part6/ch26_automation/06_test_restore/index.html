    Автоматическое тестовое восстановление¶

    Бэкап без проверки восстановления -- это не бэкап, а надежда. Единственный способ убедиться, что бэкап работает -- восстановить из него. Делать это вручную при каждом бэкапе -- нереалистично. Автоматизация -- единственный путь.

    В этой подглаве разберём, зачем нужно тестовое восстановление, какие методы песочницы существуют и как автоматизировать проверку.

    Зачем нужно автоматическое тестовое восстановление¶

    Проблема¶

    Администратор создаёт бэкапы месяцами, не проверяя их. В день катастрофы оказывается, что:

    • Архивы битые (ошибка диска, сбой при записи)
    • Отсутствуют критичные файлы (неправильный exclude)
    • Формат устарел (обновились утилиты, старый архив не читается)

    Решение¶

    Тестовое восстановление -- проверка что бэкап можно восстановить ДО реальной нужды. Автоматический скрипт регулярно разворачивает бэкап в изолированную среду, проверяет целостность и сообщает о результате.

    Важно: Цель тестового восстановления -- убедиться что бэкап рабочий без ожидания катастрофы. Это не оптимизация скорости и не шифрование -- это проверка работоспособности.

    Методы песочницы¶

    Для безопасного тестового восстановления нужны изолированные среды. Три основных метода:

    1. Проверка содержимого архива¶

    Самый простой метод — распаковать бэкап в изолированный каталог и проверить наличие ключевых файлов. Это не настоящий chroot (для него нужны /bin, /lib внутри архива), но часто достаточно просто убедиться, что файлы на месте и не повреждены.

    #!/bin/bash
# test-restore-check.sh — проверка содержимого бэкапа

BACKUP="/backup/latest.tar.gz"
SANDBOX="/tmp/restore-test"

# Создаём песочницу
rm -rf "$SANDBOX"
mkdir -p "$SANDBOX"

# Распаковываем бэкап
tar -xzf "$BACKUP" -C "$SANDBOX" || {
    echo "ОШИБКА: не удалось распаковать бэкап"
    exit 1
}

# Проверяем ключевые файлы
errors=0
for path in etc/passwd etc/fstab home; do
    if [ ! -e "${SANDBOX}/${path}" ]; then
        echo "ОШИБКА: отсутствует ${path}"
        errors=$((errors + 1))
    fi
done

# Проверка целостности (если есть хеш)
if [ -f "${BACKUP}.sha256" ]; then
    if sha256sum -c "${BACKUP}.sha256" &>/dev/null; then
        echo "Хеш: OK"
    else
        echo "ОШИБКА: хеш не совпадает"
        errors=$((errors + 1))
    fi
fi

# Очистка
rm -rf "$SANDBOX"

if [ "$errors" -gt 0 ]; then
    echo "Тест ПРОВАЛЕН: $errors ошибок"
    exit 1
fi

echo "Тест ПРОЙДЕН"
exit 0

    Примечание: Этот метод проверяет только наличие файлов, но не их работоспособность. Для полноценного тестового восстановления с запуском сервисов используйте ВМ или контейнеры.

    2. Виртуальная машина (KVM/VirtualBox)¶

    Полноценная ВМ даёт максимальную изоляцию. Можно загрузить восстановленную систему и проверить её работоспособность:

    #!/bin/bash
# test-restore-vm.sh

BACKUP="/backup/latest.tar.gz"
VM_DISK="/tmp/test-restore.img"
VM_NAME="test-restore"

# Создаём виртуальный диск
qemu-img create -f qcow2 "$VM_DISK" 20G

# Восстанавливаем бэкап на виртуальный диск
# (упрощённо: монтируем и распаковываем)
guestmount -a "$VM_DISK" -m /dev/sda1 /mnt
tar -xzf "$BACKUP" -C /mnt
guestunmount /mnt

# Запускаем ВМ для проверки
qemu-system-x86_64 \
    -drive file="$VM_DISK",format=qcow2 \
    -name "$VM_NAME" \
    -m 2048 \
    -nographic \
    -snapshot  # snapshot = без записи на диск

# Очистка
rm -f "$VM_DISK"

    3. Контейнер (Docker/LXC)¶

    Контейнеры легче ВМ, но дают хорошую изоляцию для файловой системы:

    #!/bin/bash
# test-restore-docker.sh

BACKUP="/backup/latest.tar.gz"
CONTAINER_NAME="backup-test"

# Создаём контейнер и распаковываем бэкап
docker run --rm --name "$CONTAINER_NAME" \
    -v "${BACKUP}:/backup.tar.gz:ro" \
    alpine sh -c '
    mkdir /restore
    tar -xzf /backup.tar.gz -C /restore

    # Проверяем ключевые файлы
    errors=0
    for path in etc/passwd etc/fstab; do
        if [ ! -e "/restore/$path" ]; then
            echo "MISSING: $path"
            errors=$((errors + 1))
        fi
    done

    if [ "$errors" -gt 0 ]; then
        echo "FAILED: $errors errors"
        exit 1
    fi
    echo "PASSED"
    exit 0
'

result=$?
if [ "$result" -eq 0 ]; then
    logger -t backup-test "Тестовое восстановление: ПРОЙДЕНО"
else
    logger -t backup-test "Тестовое восстановление: ПРОВАЛЕНО"
fi

    Сравнение методов¶

    Метод Изоляция Скорость Сложность Когда использовать
    Проверка содержимого Каталог Быстро Низкая Проверка файлов, хешей
    Виртуальная машина Полная Медленно Высокая Полная проверка загрузки
    Контейнер (Docker/LXC) Процесс + ФС Средне Средняя Автоматизация в CI/CD

    Важно: Прямая запись на диск -- опасный метод. Никогда не восстанавливайте бэкап на рабочий диск для проверки -- можно потерять текущие данные.

    Автоматизация тестового восстановления¶

    Полный скрипт¶

    Автоматический скрипт должен выполнять четыре шага: распаковка в песочницу -> проверка файлов -> логирование -> очистка:

    #!/bin/bash
# automated-test-restore.sh
set -euo pipefail

BACKUP_DIR="/backup"
SANDBOX="/tmp/restore-test-$$"  # $$ = PID для уникальности
LOG_TAG="backup-test"
TG_TOKEN="${TG_TOKEN:-}"
TG_CHAT_ID="${TG_CHAT_ID:-}"

log() {
    logger -t "$LOG_TAG" "$*"
    echo "[$(date '+%F %T')] $*"
}

notify() {
    local status="$1"
    local message="$2"
    if [[ -n "$TG_TOKEN" ]]; then
        local emoji="✅"
        [[ "$status" == "FAIL" ]] && emoji="❌"
        curl -sf -X POST "https://api.telegram.org/bot${TG_TOKEN}/sendMessage" \
            -d "chat_id=${TG_CHAT_ID}" \
            -d "text=${emoji} *Тест восстановления:* ${message}" \
            -d "parse_mode=Markdown" || true
    fi
}

cleanup() {
    rm -rf "$SANDBOX"
    log "Песочница очищена: $SANDBOX"
}
trap cleanup EXIT

# --- Шаг 1: Находим последний бэкап ---
LATEST=$(ls -t "${BACKUP_DIR}"/*.tar.gz 2>/dev/null | head -1)
if [ -z "$LATEST" ]; then
    log "ОШИБКА: бэкапы не найдены в $BACKUP_DIR"
    notify "FAIL" "Бэкапы не найдены"
    exit 1
fi
log "Тестируем бэкап: $LATEST"

# --- Шаг 2: Распаковка в chroot ---
mkdir -p "$SANDBOX"
log "Распаковка..."
if ! tar -xzf "$LATEST" -C "$SANDBOX" 2>/dev/null; then
    log "ОШИБКА: не удалось распаковать архив"
    notify "FAIL" "Не удалось распаковать $(basename "$LATEST")"
    exit 1
fi

# --- Шаг 3: Проверка файлов ---
log "Проверка файлов..."
errors=0

# Проверяем ключевые файлы
for path in etc/passwd etc/shadow etc/fstab; do
    if [ ! -e "${SANDBOX}/${path}" ]; then
        log "ОШИБКА: отсутствует ${path}"
        errors=$((errors + 1))
    fi
done

# Проверяем размер распакованных данных
size=$(du -sm "$SANDBOX" | cut -f1)
log "Размер распакованных данных: ${size} МБ"

if [ "$size" -lt 10 ]; then
    log "ОШИБКА: размер слишком маленький (${size} МБ) -- возможно битый архив"
    errors=$((errors + 1))
fi

# Проверяем хеш
if [ -f "${LATEST}.sha256" ]; then
    if sha256sum -c "${LATEST}.sha256" &>/dev/null; then
        log "Хеш: OK"
    else
        log "ОШИБКА: хеш не совпадает"
        errors=$((errors + 1))
    fi
fi

# --- Шаг 4: Результат ---
if [ "$errors" -gt 0 ]; then
    log "Тест ПРОВАЛЕН: $errors ошибок"
    notify "FAIL" "$(basename "$LATEST"): $errors ошибок"
    exit 1
fi

log "Тест ПРОЙДЕН: $(basename "$LATEST") (${size} МБ)"
notify "OK" "$(basename "$LATEST"): ${size} МБ, все файлы на месте"
exit 0

    Что делать если тест провалился¶

    Если тестовое восстановление провалилось, нужно выполнить три действия:

    1. Отправить алерт¶

    logger -t backup-test "КРИТИЧЕСКАЯ ОШИБКА: тестовое восстановление провалено"
curl -sf -X POST "https://api.telegram.org/bot${TG_TOKEN}/sendMessage" \
    -d "chat_id=${TG_CHAT_ID}" \
    -d "text=🔴 *КРИТИЧЕСКАЯ ОШИБКА:* Тестовое восстановление провалено! Требуется немедленное расследование." \
    -d "parse_mode=Markdown"

    2. Создать новый полный бэкап¶

    Старый бэкап может быть битым -- нужно создать новый:

    logger -t backup-test "Запускаем аварийный бэкап..."
/usr/local/bin/backup.sh

    3. Расследовать причину¶

    # Проверить логи
journalctl -t backup-test --since "1 hour ago"
journalctl -t backup --since "1 hour ago"

# Проверить диск на ошибки
dmesg | tail -50
smartctl -a /dev/sda

# Проверить архив вручную
tar -tzf /backup/latest.tar.gz | head -20

    Важно: При провале тестового восстановления правильный подход -- алерт + новый бэкап + расследование. Игнорирование, удаление всех бэкапов или перезагрузка сервера -- неправильные действия.

    Расписание тестового восстановления¶

    Тестовое восстановление должно запускаться регулярно. Рекомендуемая частота:

    Тип бэкапа Частота теста
    Ежедневный Раз в неделю
    Еженедельный Раз в месяц
    Ежемесячный Раз в квартал

    Пример cron для еженедельного теста:

    # Каждое воскресенье в 4:00 -- тестовое восстановление
0 4 * * 0 /usr/local/bin/test-restore.sh

    Или systemd timer:

    # /etc/systemd/system/test-restore.timer
[Timer]
OnCalendar=Sun *-*-* 04:00:00
Persistent=true

    Элемент Описание
