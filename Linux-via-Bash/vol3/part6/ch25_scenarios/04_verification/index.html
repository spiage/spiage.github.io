    Верификация восстановления¶

    Бэкап без проверки восстановления -- не бэкап. Это одно из главных правил системного администрирования. Вы можете годами создавать безупречные архивы, но если при катастрофе окажется, что они повреждены или неполны -- все усилия напрасны.

    Верификация -- это процесс проверки того, что: 1. Бэкап целостен (не повреждён при создании или хранении) 2. Бэкап полон (содержит все нужные данные) 3. Восстановленные данные работоспособны (система загружается, сервисы отвечают)

    Зачем нужна верификация¶

    Ситуация Что происходит без верификации Что происходит с верификацией
    Битый сектор на диске бэкапа Обнаруживаете при восстановлении в момент катастрофы Обнаруживаете сразу, пересоздаёте бэкап
    Неполный архив (исключили лишнее) Теряете данные при восстановлении Замечаете при проверке, исправляете скрипт
    Ошибка шифрования Невозможно расшифровать в критический момент Проверяете расшифровку сразу
    Сервис не запускается после восстановления Часы отладки под давлением Знаете проблему заранее

    Главный принцип: Бэкап считается успешным не когда архив создан, а когда данные успешно восстановлены и проверены.

    Методы верификации¶

    1. Сравнение хешей (sha256sum)¶

    Самый базовый метод -- сравнение криптографических хешей оригинала и восстановленного файла.

    # Создаём бэкап и сохраняем хеш
$ sha256sum /backup/home-2026-04-03.tar.gz > /backup/home-2026-04-03.tar.gz.sha256

# После восстановления -- проверяем хеш восстановленного файла
$ sha256sum /restored/home-2026-04-03.tar.gz
a1b2c3d4e5f6...  /restored/home-2026-04-03.tar.gz

# Сравниваем с оригинальным хешем
$ cat /backup/home-2026-04-03.tar.gz.sha256
a1b2c3d4e5f6...  /backup/home-2026-04-03.tar.gz

    Если хеши совпадают -- файл идентичен оригиналу. Даже один изменённый бит даст совершенно другой хеш.

    Автоматическая проверка¶

    # Проверка хеша (должен быть в том же каталоге)
$ sha256sum -c /backup/home-2026-04-03.tar.gz.sha256
/backup/home-2026-04-03.tar.gz: OK

# Если файл повреждён:
$ sha256sum -c /backup/home-2026-04-03.tar.gz.sha256
/backup/home-2026-04-03.tar.gz: FAILED
sha256sum: WARNING: 1 computed checksum did NOT match

    Проверка отдельных файлов после восстановления¶

    # Создаём хеш-файл ДО бэкапа
$ find /home/user/Documents -type f -exec sha256sum {} \; \
    > /backup/original-hashes.sha256

# После восстановления -- проверяем
$ sha256sum -c /backup/original-hashes.sha256

    2. tar -d (diff)¶

    Команда tar -d (или tar --diff) сравнивает содержимое архива с файлами на диске. Это встроенный механизм проверки tar.

    # Создаём бэкап
$ sudo tar czpf /backup/etc-2026-04-03.tar.gz /etc

# Проверяем: сравниваем архив с файловой системой
$ sudo tar df /backup/etc-2026-04-03.tar.gz

    Если tar -d не выводит ничего -- файлы идентичны. Если есть различия -- они будут показаны:

    # Пример вывода при различиях
$ sudo tar df /backup/etc-2026-04-03.tar.gz
etc/nginx/nginx.conf: Mod time differs
etc/hosts: Size differs

    Как это работает: tar -d читает архив и сравнивает каждый файл с его текущей версией на диске. Проверяются: размер, время модификации, права владельца. Не проверяется побитовое содержимое — если файл повредился (bit rot), но размер и mtime не изменились, tar -d сообщит что всё ОК. Для проверки содержимого используйте sha256sum.

    Ограничения tar -d¶

    Ограничение Описание
    Работает только с текущей ФС Сравнивает архив с текущими файлами на диске
    Не работает с сжатыми архивами напрямую Для .tar.gz нужно использовать tar df (флаг f для файла)
    Не проверяет содержимое Проверяет метаданные (размер, время), не побитовое сравнение

    3. Тестовое восстановление в песочнице¶

    Самый надёжный метод -- периодическое полное восстановление бэкапа в изолированной среде (песочнице) и проверка работоспособности.

    Восстановление в виртуальной машине¶

    # Создаём тестовую ВМ
$ virt-install \
    --name test-restore \
    --ram 2048 \
    --disk path=/var/lib/libvirt/images/test-restore.qcow2,size=40 \
    --cdrom /iso/ubuntu-24.04.iso \
    --os-variant ubuntu24.04

# Внутри ВМ: восстанавливаем бэкап
$ scp /backup/root-2026-04-03.tar.gz test-vm:/tmp/
$ sudo tar xzpf /tmp/root-2026-04-03.tar.gz -C /

# Проверяем загрузку, сервисы, данные

    Восстановление в контейнере (для файловых бэкапов)¶

    # Восстановление /etc в контейнере для проверки
$ docker run --rm -it ubuntu:24.04 bash
# Внутри контейнера:
# tar xzpf /backup/etc-2026-04-03.tar.gz
# Проверяем структуру файлов

    Восстановление в chroot¶

    # Создаём директорию для тестового восстановления
$ mkdir -p /tmp/test-restore

# Распаковываем бэкап
$ sudo tar xzpf /backup/home-2026-04-03.tar.gz -C /tmp/test-restore/

# Проверяем структуру
$ ls -la /tmp/test-restore/home/
$ find /tmp/test-restore -type f | wc -l

# Проверяем ключевые файлы
$ ls -la /tmp/test-restore/home/user/Documents/important.docx
$ file /tmp/test-restore/home/user/Documents/important.docx

    Как часто проводить тестовое восстановление¶

    Тип системы Частота Обоснование
    Критичные серверы (продакшн) Ежемесячно Высокая цена простоя
    Homelab Ежеквартально Баланс между усилиями и надёжностью
    Домашний ноутбук Раз в полгода Данные меняются нечасто
    После изменения скрипта бэкапа Сразу Убедиться, что изменения не сломали бэкап

    Золотое правило: Бэкап без проверки восстановления не является бэкапом. Минимум -- раз в квартал проводите тестовое восстановление.

    Чек-лист верификации¶

    Для файлового бэкапа¶

    #!/bin/bash
# verify-file-backup.sh -- проверка файлового бэкапа

BACKUP="$1"
RESTORE_DIR="/tmp/verify-$$"

if [ -z "$BACKUP" ]; then
    echo "Использование: $0 <backup-file>"
    exit 1
fi

echo "=== Проверка бэкапа: $BACKUP ==="

# 1. Файл существует?
if [ ! -f "$BACKUP" ]; then
    echo "FAIL: Файл не найден"
    exit 1
fi
echo "OK: Файл существует"

# 2. Проверка хеша (если есть .sha256)
if [ -f "${BACKUP}.sha256" ]; then
    if sha256sum -c "${BACKUP}.sha256" > /dev/null 2>&1; then
        echo "OK: Хеш совпадает"
    else
        echo "FAIL: Хеш НЕ совпадает!"
        exit 1
    fi
fi

# 3. Архив валиден?
if tar tzf "$BACKUP" > /dev/null 2>&1; then
    echo "OK: Архив валиден"
else
    echo "FAIL: Архив повреждён!"
    exit 1
fi

# 4. Тестовое восстановление
mkdir -p "$RESTORE_DIR"
tar xzpf "$BACKUP" -C "$RESTORE_DIR"
FILE_COUNT=$(find "$RESTORE_DIR" -type f | wc -l)
echo "OK: Восстановлено файлов: $FILE_COUNT"

# 5. Проверка ключевых файлов (адаптируйте под себя)
for path in "home/user/Documents" "home/user/.config" "etc/nginx"; do
    if [ -e "${RESTORE_DIR}/${path}" ]; then
        echo "OK: ${path} существует"
    else
        echo "WARN: ${path} отсутствует"
    fi
done

# 6. Очистка
rm -rf "$RESTORE_DIR"

echo "=== Проверка завершена ==="

    Для бэкапа базы данных¶

    #!/bin/bash
# verify-db-backup.sh -- проверка дампа БД

BACKUP="$1"
DB_TYPE="${2:-postgres}"  # postgres или mysql

echo "=== Проверка дампа БД: $BACKUP ==="

# 1. Файл существует и не пуст?
if [ ! -s "$BACKUP" ]; then
    echo "FAIL: Файл пуст или не найден"
    exit 1
fi
echo "OK: Файл существует ($(du -h "$BACKUP" | cut -f1))"

# 2. Дамп валиден? (проверка первых строк)
if [ "$DB_TYPE" = "postgres" ]; then
    if gunzip -c "$BACKUP" | head -20 | grep -q "PostgreSQL"; then
        echo "OK: Дамп PostgreSQL валиден"
    else
        echo "WARN: Не удалось подтвердить формат дампа"
    fi
elif [ "$DB_TYPE" = "mysql" ]; then
    if gunzip -c "$BACKUP" | head -20 | grep -q "MySQL"; then
        echo "OK: Дамп MySQL валиден"
    else
        echo "WARN: Не удалось подтвердить формат дампа"
    fi
fi

# 3. Тестовое восстановление в тестовую БД
echo "Тестовое восстановление..."
if [ "$DB_TYPE" = "postgres" ]; then
    createdb test_restore_verify
    gunzip -c "$BACKUP" | psql -U postgres test_restore_verify
    TABLE_COUNT=$(psql -U postgres -t -c "SELECT count(*) FROM information_schema.tables WHERE table_schema = 'public';" test_restore_verify)
    echo "OK: Таблиц восстановлено: $TABLE_COUNT"
    dropdb test_restore_verify
fi

echo "=== Проверка завершена ==="

    Для Docker-бэкапа¶

    #!/bin/bash
# verify-docker-backup.sh -- проверка Docker-бэкапа

VOLUME_BACKUP="$1"
TEST_VOLUME="verify-test-$$"

echo "=== Проверка Docker volume: $VOLUME_BACKUP ==="

# 1. Создаём тестовый volume
docker volume create "$TEST_VOLUME"

# 2. Восстанавливаем
docker run --rm \
    -v "${TEST_VOLUME}:/data" \
    -v "$(dirname "$VOLUME_BACKUP"):/backup" \
    alpine tar xzf "/backup/$(basename "$VOLUME_BACKUP")" -C /data

# 3. Проверяем содержимое
FILE_COUNT=$(docker run --rm \
    -v "${TEST_VOLUME}:/data" \
    alpine find /data -type f | wc -l)
echo "OK: Восстановлено файлов: $FILE_COUNT"

# 4. Проверяем ключевые файлы (адаптируйте)
docker run --rm \
    -v "${TEST_VOLUME}:/data" \
    alpine ls -la /data/

# 5. Очистка
docker volume rm "$TEST_VOLUME"

echo "=== Проверка завершена ==="

    Проверка работоспособности сервисов¶

    После восстановления системы или сервера нужно проверить, что сервисы работают:

    HTTP/HTTPS сервисы¶

    # Проверка HTTP
$ curl -s -o /dev/null -w "HTTP %{http_code}, время: %{time_total}s" http://localhost/
HTTP 200, время: 0.012s

# Проверка HTTPS (с игнорированием самоподписанных сертификатов)
$ curl -sk -o /dev/null -w "HTTP %{http_code}" https://localhost/
HTTP 200

# Проверка конкретного endpoint
$ curl -s http://localhost:8080/health | jq .
{
  "status": "healthy",
  "database": "connected",
  "uptime": 3600
}

    Метод Что проверяет Автоматизация Надёжность
    sha256sum Целостность файла (бит-в-бит) Полная Высокая
    tar -d Соответствие архива файловой системе Полная Средняя
    Тестовое восстановление Полная работоспособность Частичная Максимальная

    Типичные ошибки верификации¶

    Ошибка Последствие Как избежать
    Проверка только размера файла Повреждённый файл того же размера пройдёт Используйте sha256sum
    Отсутствие тестового восстановления Бэкап может быть непригоден для восстановления Восстанавливайте в песочнице регулярно
    Проверка только при создании бэкапа Бэкап может повредиться при хранении Периодически перепроверяйте старые бэкапы
    Визуальная проверка файлов Ненадёжна, невозможно автоматизировать Используйте хеши и автоматические тесты
    Проверка «бэкап создался без ошибок» Ошибка создания != пригодность восстановления Всегда проверяйте восстановление отдельно

    Автоматизация верификации через cron¶

    #!/bin/bash
# verify-all-backups.sh -- автоматическая проверка всех бэкапов

BACKUP_DIR="/backup"
LOG_FILE="/var/log/backup-verify.log"
ERRORS=0

echo "=== $(date): Проверка бэкапов ===" >> "$LOG_FILE"

# Проверка хешей
for hash_file in "$BACKUP_DIR"/*.sha256; do
    if [ -f "$hash_file" ]; then
        if ! sha256sum -c "$hash_file" >> "$LOG_FILE" 2>&1; then
            echo "ERROR: Хеш не совпадает: $hash_file" >> "$LOG_FILE"
            ERRORS=$((ERRORS + 1))
        fi
    fi
done

# Проверка валидности архивов
for archive in "$BACKUP_DIR"/*.tar.gz; do
    if [ -f "$archive" ]; then
        if ! tar tzf "$archive" > /dev/null 2>&1; then
            echo "ERROR: Повреждён архив: $archive" >> "$LOG_FILE"
            ERRORS=$((ERRORS + 1))
        fi
    fi
done

if [ "$ERRORS" -eq 0 ]; then
    echo "OK: Все бэкапы прошли проверку" >> "$LOG_FILE"
else
    echo "ERROR: $ERRORS бэкапов не прошли проверку!" >> "$LOG_FILE"
    # Отправляем уведомление (email, telegram, etc.)
    echo "ВНИМАНИЕ: $ERRORS бэкапов повреждены!" | mail -s "Backup Alert" admin@example.com
fi
