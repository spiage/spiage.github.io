От основ Linux к мастерству Bash

Защита адресного пространства процессов

Каждый процесс имеет изолированное виртуальное адресное пространство.

Виртуальная память

Виртуальная память — абстракция, создающая иллюзию独占ной памяти для каждого процесса.

Процесс A:          Процесс B:
0x0000 ────────     0x0000 ────────
│ свой код   │     │ свой код   │
│ свои данные│     │ свои данные│
│ свой стек  │     │ свой стек  │
0xFFFF ────────     0xFFFF ────────

Физическая память:
┌─────────────────┐
│ Страницы A      │
├─────────────────┤
│ Страницы B      │
├─────────────────┤
│ Ядро            │
└─────────────────┘

Ключевой момент: оба процесса «видят» всю память от 0x0000 до 0xFFFF, но физически находятся в разных местах.

MMU (Memory Management Unit)

MMU — аппаратный компонент процессора для трансляции виртуальных адресов в физические.

Как работает MMU

Процесс: "читать адрес 0x12345000"
    ↓
MMU: смотрит в таблице страниц
    ↓
Физический адрес: 0x87650000
    ↓
RAM: чтение данных

Таблица страниц

Ядро хранит таблицу трансляции для каждого процесса:

// Упрощённо
struct page_table {
    virtual_addr: 0x12345000,
    physical_addr: 0x87650000,
    permissions: R/W/X,  // Read/Write/Execute
    user_accessible: true/false
};

Проверка прав:

  • Если процесс пытается читать память без права R → Page Fault
  • Если пытается писать без права W → Page Fault
  • Если пытается выполнять без права X → Page Fault

Защита между процессами

ДействиеРезультат
Процесс A читает свою память✅ Успех
Процесс A читает память процесса B❌ Segmentation fault
Процесс A читает память ядра❌ Segmentation fault
Процесс в user mode обращается к I/O порту❌ General protection fault

Почему это работает

  1. Разные таблицы страниц у каждого процесса
  2. MMU проверяет каждый доступ к памяти
  3. Page Fault при нарушении прав

Segmentation Fault

SIGSEGV — сигнал, который ядро посылает процессу при попытке доступа к недопустимой памяти.

Причины

# 1. Разыменование нулевого указателя
NULL = None
*NULL = 42  # Segmentation fault

# 2. Чтение за пределами выделенной области
int arr[10];
arr[100] = 42;  # Segmentation fault

# 3. Запись в страницу только для чтения
char *str = "hello";
str[0] = 'H';  # Segmentation fault

В bash

# bash не позволяет напрямую работать с памятью
# Но можно вызвать программу с ошибкой

# Пример: segfault в C программе
cat > test.c << 'EOF'
int main() {
    int *ptr = 0;
    *ptr = 42;  // Segfault
}
EOF

gcc test.c -o test
./test
# Segmentation fault (core dumped)

Обработка SIGSEGV

# Ловушка на segfault (не всегда работает)
trap 'echo "Segfault caught!"' SEGV

# Запустить программу
./crashy_program

Почему это важно для bash

Переменные не экспортируются обратно

# Переменная в подпроцессе
bash -c 'FOO=bar'

# FOO не видна в родительском процессе
echo $FOO  # Пусто!

# Почему: у каждого процесса своя память
# bash (родитель) ─fork─→ bash (дочерний)
#   FOO не существует      FOO=bar (в своей памяти)

Копирование при записи (Copy-on-Write)

# fork() не копирует память сразу
# Дочерний процесс разделяет страницы с родителем

# При записи — копия страницы
bash -c 'VAR=new_value'  # Копия страницы с VAR

Проверка:

# Посмотреть использование памяти
ps -o pid,rss,vsz,comm

# RSS (Resident Set Size) — физическая память
# VSZ (Virtual Size) — виртуальная память

Инструменты наблюдения

/proc//maps

# Карта памяти процесса
cat /proc/$$/maps

# Вывод:
# 555555554000-555555555000 r-xp 00000000 08:01 123456 /bin/bash
# 555555754000-555555755000 r--p 00001000 08:01 123456 /bin/bash
# 555555755000-555555756000 rw-p 00002000 08:01 123456 /bin/bash
# 7ffff7a00000-7ffff7b9d000 r-xp 00000000 08:01 789012 /lib/x86_64-linux-gnu/libc-2.31.so
# 7ffffffde000-7ffffffff000 rw-p 00000000 00:00 0 [stack]

Колонки:

  • Адрес начала-конца
  • Права (r=read, w=write, x=execute, p=private)
  • Смещение в файле
  • Устройство:inode
  • Путь к файлу

/proc//status

# Использование памяти
cat /proc/$$/status | grep -E "^(VmSize|VmRSS|VmData|VmStk):"

# Вывод:
# VmSize:    100000 kB  # Виртуальная память
# VmRSS:      10000 kB  # Физическая память ( Resident )
# VmData:      5000 kB  # Данные (heap)
# VmStk:        136 kB  # Стек

pmap

# Детальная карта памяти
pmap -x $$

# Вывод:
# Address           Kbytes     RSS   Dirty Mode  Mapping
# 0000555555554000     136      40       0 r-x-- bash
# 00007ffff7a00000    1648     800       0 r-x-- libc-2.31.so
# 00007ffffffde000     132      20      20 rw---   [ stack ]

Практические примеры

Утечка памяти

#!/bin/bash

# Мониторинг памяти процесса
PID=$$

while true; do
    RSS=$(grep VmRSS /proc/$PID/status | awk '{print $2}')
    echo "$(date): RSS = ${RSS} kB"
    sleep 5
done

Проверка доступа к памяти

#!/bin/bash

# Попытка прочитать память другого процесса
read_process_memory() {
    local pid=$1
    local addr=$2
    
    # Требуется ptrace capability
    if ! grep -q ptrace /proc/$pid/status 2>/dev/null; then
        echo "Cannot access memory of $pid"
        return 1
    fi
}

Ограничение памяти

#!/bin/bash

# Ограничить память процесса
ulimit -v 100000  # 100MB виртуальной памяти

# Запустить команду
memory_hungry_command

# Или через cgroup
cgcreate -g memory:/limited
cgset -r memory.limit_in_bytes=100M limited
cgexec -g memory:limited memory_hungry_command
