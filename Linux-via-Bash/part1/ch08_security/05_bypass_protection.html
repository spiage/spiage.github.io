Обход защиты

Методы эксплуатации уязвимостей несмотря на защиты.

Почему защиты не абсолютны

Защиты повышают стоимость атаки, но не делают её невозможной:

Защита → Контрмера → Новая защита → Новый обход

ROP (Return-Oriented Programming)

Что это

ROP использует существующий код в памяти для обхода DEP/NX.

Вместо:                 ROP:
┌─────────────┐        ┌─────────────┐
│ Shellcode   │        │ gadget 1    │ → pop rdi; ret
├─────────────┤        ├─────────────┤
│ Запрещён NX │        │ gadget 2    │ → pop rsi; ret
└─────────────┘        ├─────────────┤
                       │ gadget 3    │ → call system
                       └─────────────┘

Как работает

  1. Найти гаджеты в libc/бинарнике
  2. Построить цепочку в стеке
  3. Переполнить буфер
  4. Выполняется цепочка гаджетов

Пример (концептуально)

# ROP chain для system("/bin/sh")
payload = b"A" * 64  # Заполнение
payload += p64(pop_rdi)      # Гаджет 1
payload += b"/bin/sh\x00"    # Аргумент
payload += p64(system)       # Вызов

Защита

  • ASLR — рандомизирует адреса гаджетов
  • CFI — проверяет поток выполнения

return-to-libc

Что это

return-to-libc вызывает существующие функции libc вместо shellcode.

Стек:
┌─────────────────┐
│ ...             │
├─────────────────┤
│ адрес system    │ ← Вместо return
├─────────────────┤
│ адрес exit      │ ← После system
├─────────────────┤
│ "/bin/sh"       │ ← Аргумент
└─────────────────┘

Пример (концептуально)

payload = b"A" * 64
payload += p64(system_addr)    # Вызов system
payload += p64(exit_addr)      # После возврата
payload += p64(binsh_addr)     # Аргумент "/bin/sh"

Защита

  • ASLR — рандомизирует адреса libc
  • RELRO — защищает GOT таблицу
  • Stack Canary — обнаруживает переполнение

Spectre и Meltdown

Что это

Атаки по сторонним каналам через спекулятивное выполнение.

Meltdown (CVE-2017-5754)

Читает память ядра из user space:

1. Ядро мапит память в user space
2. Спекулятивное выполнение читает ядро
3. Данные остаются в кэше
4. Side-channel атака восстанавливает данные

Затронуты: Intel (частично), некоторые ARM

Spectre v1 (CVE-2017-5753)

Обход проверки границ:

1. if (x < array_size) {
2.     value = array[x];  // Спекулятивно выполняется
3. }
4. // x может быть > array_size
5. // Но значение прочитано в кэш

Затронуты: Все современные CPU

Spectre v2 (CVE-2017-5715)

Отравление branch predictor:

1. Тренировка predictor'а
2. Жертва выполняет неправильный переход
3. Спекулятивное чтение данных

Проверка уязвимости

# Проверить статус Meltdown/Spectre
grep . /sys/devices/system/cpu/vulnerabilities/*

# Вывод:
# Meltdown: Mitigation: PTI
# Spectre v1: Mitigation: load_fences
# Spectre v2: Mitigation: retpoline

Защита

УязвимостьМитигация
MeltdownKPTI/KAISER (разделение страниц)
Spectre v1load_fences, компилятор
Spectre v2retpoline, микрокод

Влияние на производительность

МитигацияНакладные расходы
KPTI5-30% (I/O интенсивные)
Retpoline2-5%
load_fences<1%

L1TF / Foreshadow

Что это

L1 Terminal Fault — чтение памяти через кэш L1.

1. Инвалидация TLB
2. Спекулятивное выполнение
3. Чтение из L1 кэша
4. Восстановление данных

Затронуты: Intel (2011+), некоторые AMD

Защита

  • Микрокод CPU
  • Отключение SMT/Hyper-Threading
  • KVM flush

Практические примеры

Проверка митигаций

#!/bin/bash

echo "=== Vulnerability Status ==="

for file in /sys/devices/system/cpu/vulnerabilities/*; do
    name=$(basename "$file")
    status=$(cat "$file")
    printf "%-20s %s\n" "$name:" "$status"
done

Аудит бинарников

#!/bin/bash

# Проверить защиты бинарника
check_binary() {
    local bin="$1"
    
    echo "=== $bin ==="
    
    # PIE
    if file "$bin" | grep -q "pie"; then
        echo "PIE: ✓"
    else
        echo "PIE: ✗"
    fi
    
    # RELRO
    if readelf -l "$bin" 2>/dev/null | grep -q "GNU_RELRO"; then
        if readelf -l "$bin" 2>/dev/null | grep -q "BIND_NOW"; then
            echo "Full RELRO: ✓"
        else
            echo "Partial RELRO: ✓"
        fi
    else
        echo "RELRO: ✗"
    fi
    
    # Stack canary
    if nm "$bin" 2>/dev/null | grep -q "__stack_chk"; then
        echo "Stack Canary: ✓"
    else
        echo "Stack Canary: ✗"
    fi
    
    # NX
    if readelf -l "$bin" 2>/dev/null | grep "GNU_STACK" | grep -q "RW"; then
        echo "NX: ✗ (executable stack)"
    else
        echo "NX: ✓"
    fi
}

check_binary /bin/bash

Рекомендации

  1. Включите все защиты на уровне системы
  2. Компилируйте с защитами (PIE, canary, RELRO)
  3. Обновляйте микрокод CPU
  4. Минимизируйте поверхность атаки
  5. Мониторьте аномалии
