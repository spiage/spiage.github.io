От основ Linux к мастерству Bash

Буферизация и кэширование в ядре

Ядро кэширует данные для повышения производительности.

Page Cache

Page Cache — кэш страниц памяти для данных с диска.

Диск ←→ Page Cache ←→ Приложение
         (ядро)

Преимущества

  1. Быстрое чтение: повторное чтение из памяти, а не с диска
  2. Отложенная запись: запись в кэш, сброс позже
  3. Объединение операций: несколько записей в одну

Последствия для скриптов

# Запись в файл
echo "data" > file.txt
# Данные в Page Cache, но ещё не на диске!

# Чтение сразу после записи
cat file.txt  # Читает из кэша — быстро

# Внезапное отключение питания — данные потеряны

Проверка состояния кэша

# Общая информация
free -h
#              total        used        free
# Mem:           15G         8G         2G
# Buff/cache:    5G

# Детальная информация
cat /proc/meminfo | grep -E '^(Cached|Dirty|Writeback)'
# Cached: 5000000 kB      # В кэше
# Dirty: 1000 kB          # Ждут записи на диск
# Writeback: 0 kB         # Записываются прямо сейчас

Синхронизация с диском

sync — сброс кэша

# Сбросить все кэши на диск
sync

# Подождать завершения сброса
sync && echo "Данные на диске"

Что делает sync:

  1. Записывает dirty страницы на диск
  2. Обновляет метаданные ФС
  3. Возвращает когда все записано

fsync — сброс конкретного файла

// C пример
int fd = open("file", O_WRONLY);
write(fd, "data", 4);
fsync(fd);  // Гарантировать запись на диск
close(fd);

В bash:

# dd с синхронизацией
dd if=data of=file conv=fsync

# Или после записи
echo "data" > file && sync

Буферизация записи

Write-back caching

# Запись
echo "data" > file

# Что происходит:
# 1. Данные копируются в Page Cache
# 2. syscall write() завершается успешно
# 3. Ядро позже сбрасывает на диск (через 5-30 сек)

Проверка:

# Записать и проверить Dirty
echo "test" > /tmp/testfile
cat /proc/meminfo | grep Dirty

# Подождать и проверить снова
sleep 10
cat /proc/meminfo | grep Dirty

Прямой I/O (без кэша)

# dd без кэширования
dd if=data of=file oflag=direct

# Или с O_DIRECT
dd if=data of=file conv=notrunc oflag=direct

Когда использовать:

  • Большие последовательные записи
  • Данные не понадобятся снова
  • Нужно гарантировать запись

Мониторинг кэша

free

# Статистика памяти
free -h

# Вывод:
#               total        used        free      shared  buff/cache   available
# Mem:           15Gi       4.0Gi       2.0Gi       500Mi       9.0Gi        10Gi
# Swap:         8.0Gi          0B       8.0Gi

Колонки:

  • buff/cache — Page Cache + буферы
  • available — Сколько можно выделить приложениям

vmstat

# Статистика в реальном времени
vmstat 1

# Вывод:
# procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
#  r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
#  1  0      0 2048000 256000 4096000    0    0    10    20  100  200 10  2 88  0  0

Колонки memory:

  • swpd — Использовано swap
  • free — Свободная память
  • buff — Буферы ядра
  • cache — Page Cache

iostat

# Статистика I/O
iostat -x 1

# Вывод:
# Device            r/s     w/s     rkB/s     wkB/s   %util
# sda              10.00   20.00    100.00    200.00   15.00

%util > 80% — диск перегружен!

Влияние на производительность

Первое чтение vs повторное

# Очистить кэш (требует root)
echo 3 > /proc/sys/vm/drop_caches

# Первое чтение — с диска
time cat large_file.iso  # Медленно (10 сек)

# Повторное чтение — из кэша
time cat large_file.iso  # Быстро (0.1 сек, в 100 раз!)

Write buffering

# Много маленьких записей
for i in {1..1000}; do
    echo "line $i" >> file.txt  # Каждый раз в кэш
done

# Одна большая запись
{
    for i in {1..1000}; do
        echo "line $i"
    done
} > file.txt  # Один syscall, меньше накладных расходов

Потеря данных при сбое

Проблема

# Скрипт записывает важные данные
echo "critical config" > /etc/myapp.conf
# Данные в кэше...

# Внезапное отключение питания
# Данные потеряны!

Решение

# Гарантированная запись
{
    echo "critical config"
} > /etc/myapp.conf && sync

# Или использовать dd
dd if=/dev/stdin of=/etc/myapp.conf conv=fsync <<EOF
critical config
EOF

Журналируемые ФС

Современные ФС (ext4, XFS, Btrfs) используют журналирование:

# Проверить тип ФС
df -T /home
# Filesystem     Type
# /dev/sda1      ext4

# ext4 журналирует метаданные по умолчанию
# Данные могут потерять, но ФС останется целой

Direct I/O vs Buffered I/O

РежимПреимуществаНедостатки
BufferedБыстро, кэшированиеДанные не сразу на диске
DirectГарантия записиМедленнее, нет кэша

Когда использовать Direct I/O

# Базы данных (свой кэш)
dd if=/dev/sda of=backup.img oflag=direct

# Большие файлы (не понадобятся снова)
dd if=source.iso of=dest.iso oflag=direct bs=4M

# Стриминг (данные не кэшируются)
dd if=/dev/video0 of=stream.ts oflag=direct

Практические рекомендации

1. Используйте sync после критичных записей

#!/bin/bash

# Записать конфиг
cp new.conf /etc/myapp.conf

# Гарантировать запись
sync

# Или для конкретного файла
xfs_syncfs /etc/myapp.conf  # Для XFS

2. Группируйте записи

#!/bin/bash

# ПЛОХО: много мелких записей
for line in $(cat data); do
    echo "$line" >> output.txt
done

# ХОРОШО: одна запись
cat data >> output.txt

3. Мониторьте Dirty страницы

#!/bin/bash

# Проверить очередь записи
DIRTY=$(grep Dirty /proc/meminfo | awk '{print $2}')

if [[ $DIRTY -gt 1000000 ]]; then
    echo "Warning: High dirty pages ($DIRTY kB)"
fi

4. Избегайте O_DIRECT без необходимости

# ПЛОХО: без причины
dd if=file of=/dev/null oflag=direct  # Медленнее!

# ХОРОШО: с причиной
dd if=file of=/dev/null  # Использует кэш

5. Понимайте задержку записи

#!/bin/bash

# По умолчанию ядро сбрасывает через 30 сек
cat /proc/sys/vm/dirty_expire_centisecs
# 3000 (30 секунд)

# Можно изменить
echo 500 > /proc/sys/vm/dirty_expire_centisecs  # 5 секунд
