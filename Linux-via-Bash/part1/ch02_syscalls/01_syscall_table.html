Таблица системных вызовов и контракт между shell и ядром

Системные вызовы — единственный способ взаимодействия с ядром.

Что такое syscall interface

Системный вызов (syscall) — запрограммированный вход в ядро для выполнения привилегированной операции.

Аналогия:

Приложение → Официант → Кухня
     ↓          ↓         ↓
  User space  Syscall  Kernel space

Вы не можете попасть на кухню (ядро) напрямую — только через официанта (syscall).

Как bash использует syscalls

Каждая команда bash транслируется в последовательность syscall’ов:

Пример 1: echo

echo "hello"

Что происходит:

1. write(1, "hello\n", 7) = 7    # Запись в stdout
2. exit_group(0)                 # Завершение

Проверка через strace:

strace -e trace=write,exit_group bash -c 'echo "hello"'
# write(1, "hello\n", 7) = 7
# exit_group(0) = ?

Пример 2: cat file.txt

cat file.txt

Что происходит:

1. openat(AT_FDCWD, "file.txt", O_RDONLY) = 3
2. read(3, "content\n", 65536) = 8
3. write(1, "content\n", 8) = 8
4. read(3, "", 65536) = 0         # EOF
5. close(3) = 0
6. exit_group(0) = ?

Проверка через strace:

strace -e trace=openat,read,write,close cat file.txt

Пример 3: ls -l

ls -l

Что происходит:

1. openat(AT_FDCWD, ".", O_RDONLY|O_NONBLOCK|O_DIRECTORY) = 3
2. getdents64(3, ..., 32768) = 144    # Чтение директории
3. newfstatat(3, "file1", ...) = 0    # Получение метаданных
4. newfstatat(3, "file2", ...) = 0
5. write(1, "total 4\n...", 100) = 100
6. close(3) = 0

Пример 4: Перенаправление

echo "data" > file.txt

Что происходит:

1. openat(AT_FDCWD, "file.txt", O_WRONLY|O_CREAT|O_TRUNC, 0644) = 3
2. write(3, "data\n", 5) = 5
3. close(3) = 0

Важно: > — это не команда, это shell обрабатывает редирект до exec!

Пример 5: Конвейер

cat file.txt | grep "pattern"

Что происходит:

1. pipe() = [4, 5]                    # Создание pipe

# Процесс 1 (cat)
2. fork() = 100
3. close(4)                           # Закрыть read end
4. dup2(5, 1)                         # stdout → pipe write
5. openat(..., "file.txt", ...) = 6
6. sendfile(5, 6, ...)                # Запись в pipe
7. close(5)

# Процесс 2 (grep)
8. fork() = 101
9. close(5)                           # Закрыть write end
10. dup2(4, 0)                        # stdin ← pipe read
11. read(0, ..., 65536)               # Чтение из pipe
12. write(1, "match\n", 6) = 6

Практическое значение

1. Отладка

# Почему команда не работает?
strace command 2>&1 | grep -i error

# Какой файл не найден?
strace -e trace=openat command 2>&1 | grep ENOENT

2. Производительность

# Много мелких syscall?
strace -c command

# Решение: буферизация, пакетная обработка

3. Безопасность

# Какие syscall использует программа?
strace -c program

# Ограничить через seccomp
seccomp-bpf filter

4. Понимание поведения

# Что делает cd?
strace -e trace=chdir bash -c 'cd /tmp'
# chdir("/tmp") = 0

# cd — builtin, не fork/exec!

Встроенные команды vs внешние

Встроенные (builtin)

# Не создают новый процесс
cd /tmp          # syscall: chdir()
export VAR=val   # syscall: нет (переменная shell)
echo $VAR        # syscall: write()

Проверка:

type cd
# cd is a shell builtin

strace -e trace=fork,execve bash -c 'cd /tmp'
# Нет fork/execve!

Внешние команды

# Создают новый процесс
ls /tmp          # fork + execve + wait
cat file.txt     # fork + execve + wait

Проверка:

type ls
# ls is aliased to 'ls --color=auto'

strace -e trace=fork,execve bash -c 'ls /tmp'
# execve("/bin/ls", ...) = 0

Оптимизация в скриптах

# ПЛОХО: много syscall в цикле
for file in *.txt; do
    wc -l "$file"    # fork + exec для каждого файла
done

# ХОРОШО: один syscall
wc -l *.txt          # Один процесс обрабатывает все

Буферизация

# read() по одному байту (медленно)
while read -n1 char; do
    ...
done < file

# read() буфером (быстро)
while read line; do
    ...
done < file
